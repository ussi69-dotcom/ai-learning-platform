## Section 0: Before We Begin

Welcome! This section is for complete beginners. If you already have your development environment set up, feel free to skip ahead.

### What is Claude Code?
Claude Code is a tool that runs in your terminal (command line). Unlike chat interfaces like ChatGPT or web-based Claude, Claude Code has direct access to files on your computer. It can read, write, and edit code, run commands, and collaborate with you on real projects.

### Why Terminal Instead of Chat?
- **File access:** It can read your entire project and understand context.
- **Command execution:** It can install dependencies, run tests, and build your application.
- **Automation:** Enables complex workflows that aren't possible in chat.

Don't worry if the terminal is new to you. We'll walk you through the basics.

### What You'll Need:
1.  **A Computer:** with Windows, macOS, or Linux.
2.  **Node.js and npm:** JavaScript runtime environment. We'll show you how.
3.  **A Terminal:** An application for typing commands.
4.  **Anthropic API Key:** For accessing Claude models. Get it at [anthropic.com/console](https://console.anthropic.com/).

## Section 1: Installation and Setup

Let's set up everything you need together. This process differs slightly depending on your operating system.

<Diagram type="claude-code-installation-flow" />

### 1.1 For Windows Users: WSL (Optional but Recommended)

<Callout type="info">
**What is WSL?** Windows Subsystem for Linux lets you run Linux directly inside Windows. It's the preferred way to work with modern development tools on Windows.
</Callout>

**Installing WSL2:**

1. Open **PowerShell as Administrator** (right-click ‚Üí Run as administrator)
2. Enter the command:
   ```powershell
   wsl --install
   ```
3. Restart your computer
4. After restart, Ubuntu will open automatically and prompt you to create a username and password

<Callout type="tip">
**VS Code Integration:** Install the "WSL" extension in VS Code. Then you can open any folder in WSL using "Open Folder in WSL".
</Callout>

### 1.2 Terminal: Your Gateway to Efficiency

The terminal (or command line) is a text interface for controlling your computer. Instead of clicking icons, you type commands. It's an incredibly powerful tool for developers.

- **On macOS:** Find it as `Terminal.app`.
- **On Linux:** Usually called `Terminal` or `Console`.
- **On Windows with WSL:** Open Ubuntu from the Start menu.

<Diagram type="terminal-basics" />

### 1.3 Installing Node.js and Claude Code

Node.js is an environment that allows you to run JavaScript code outside the browser. `npm` is its package manager.

**Step 1: Install Node.js**

The easiest way is to download the installer from [nodejs.org](https://nodejs.org/). We recommend the **LTS (Long-Term Support)** version.

For Linux/WSL, you can use:
```bash
curl -fsSL https://deb.nodesource.com/setup_lts.x | sudo -E bash -
sudo apt-get install -y nodejs
```

**Step 2: Verify Installation**

Open your terminal and type:
```bash
node -v
npm -v
```
You should see version numbers for both tools.

**Step 3: Install Claude Code CLI**

```bash
npm install -g @anthropic-ai/claude-code
```

**Step 4: Set Up Your API Key**

Claude will ask for your Anthropic API key on first launch. Have it ready from [console.anthropic.com](https://console.anthropic.com/).

**Step 5: Verify Installation**

```bash
claude --version
```

<Callout type="success">
**Done!** Your environment is ready. You're just a few steps away from letting AI work for you.
</Callout>

## Section 2: The PSB System

Most developers fail with AI agents because they skip the most critical step: **context**. They jump straight into "Build" without the "Plan" or "Setup."

The **PSB System** (Plan, Setup, Build) fixes this by enforcing a high-success framework.

<Diagram type="claude-code-psb-workflow" />

### Phase 1: PLAN ‚Äî The Blueprint

Before writing a single line of code, align the agent on *what* you're building. Create a **Project Spec Doc**.

**What to include:**
- **Goal:** What problem are we solving?
- **Constraints:** Time, tech stack, dependencies
- **Milestones:** Deliverable checkpoints
- **Success criteria:** How do we know it works?

### Phase 2: SETUP ‚Äî The Constitution

This is where the magic happens. Create a "Constitution" for your project‚Äîa `CLAUDE.md` file with rules that Claude must follow.

**Example `CLAUDE.md`:**

```markdown
# Project: Auth System
## Tech Stack
- Next.js 14 (App Router)
- Tailwind CSS
- Supabase (Auth + DB)

## Code Style
- TypeScript strict mode
- Functional components only
- File naming: kebab-case

## Rules
- NEVER delete code without explaining why
- ALWAYS run tests before committing
- Limit file changes to <100 lines per turn

## Security
- NO hardcoded secrets
- Review all API endpoints for auth checks
```

Each section of your CLAUDE.md serves a specific purpose:

<Diagram type="claude-md-anatomy" />

### Phase 3: BUILD ‚Äî The Execution

Now, and only now, do you build. You use **Plan Mode** for complex features and let Claude delegate to **Sub-agents**.

<Callout type="tip">
**Pro tip:** The more context in CLAUDE.md, the fewer mid-task corrections you'll need. Invest 10 minutes upfront, save hours later.
</Callout>

## Section 4: Core Features

You know the rules. Now let's look at the weapons in your arsenal.

### Plan Mode ‚Äî Think Before Acting

Plan Mode is the difference between a toy and a tool. When you enter Plan Mode, Claude doesn't just react‚Äîit **analyzes**. It scans your codebase and proposes a comprehensive implementation plan.

**How to enter Plan Mode:**
- Type `/plan` before your prompt

<Diagram type="plan-mode-flow" />

**When to use Plan Mode:**
- Multi-file refactors
- New feature implementations
- Architectural changes
- When you're unsure of the approach

### Essential Commands

| Command | Purpose | When to Use |
|---------|---------|-------------|
| `/help` | List available commands | Discovery |
| `/clear` | Reset conversation | Starting new task |
| `/model` | Switch models | Speed vs quality tradeoff |
| `/plan` | Enter Plan Mode | Complex features |
| `/cost` | Check token usage | Budget awareness |

### Sub-Agent Architecture

For complex tasks, Claude spawns **specialized sub-agents**:

<Diagram type="sub-agent-architecture" />

| Agent | Purpose |
|-------|---------|
| **Explore** | Codebase search, pattern matching |
| **Plan** | Architecture design, implementation planning |
| **Review** | Code quality, security checks |
| **Test** | Test execution, verification |

**Key insight:** Each sub-agent gets a **forked context**. Their work doesn't pollute your main conversation.

## Section 5: Advanced Features

These features separate casual users from power users.

### MCP Servers ‚Äî External Superpowers

MCP (Model Context Protocol) lets Claude connect to external tools and data sources.

<Diagram type="claude-code-ecosystem" />

**Popular MCP Servers:**

| Server | Capability | Use Case |
|--------|------------|----------|
| `github` | PRs, issues, code review | "Create PR for issue #42" |
| `playwright` | Browser automation | Visual testing |
| `postgres` | Database queries | "What's in the users table?" |
| `context7` | Library documentation | Up-to-date API reference |

**Adding an MCP Server:**

```bash
# List available servers
claude mcp list

# Add a server
claude mcp add github
```

### Hooks ‚Äî Automated Workflows

Hooks execute shell commands in response to Claude Code events.

**Common use cases:**
- Auto-format code after writes
- Run linter after changes
- Execute tests after file modifications

<Callout type="warning">
**Security Warning:** Hooks execute shell commands! Review carefully before enabling.
</Callout>

## Troubleshooting

| Problem | Solution |
|---------|----------|
| Claude loops on same error | `/clear` and rephrase with more context |
| Wrong assumptions made | "Stop. List your assumptions. X is incorrect because..." |
| Too many files changed | "Revert last change. Only modify [specific file]" |
| Context overflow | `/compact` or start fresh with `/clear` |
| Hallucinated API/method | "Verify this API exists. Show me the documentation." |
| Permission denied errors | Check file permissions, use `sudo` if needed |

### Context Management Strategy

One of the most common issues is **context overflow**. Understanding when to compact vs. clear is key:

### Security Best Practices

1. **Review diffs** before approving
2. **Git commit frequently** for rollback
3. **Test in branches** ‚Äî never push to main directly
4. **Audit hooks** ‚Äî untrusted hooks are security risks
5. **Limit scope** ‚Äî "Only touch files in src/auth/"
